# 项目问题总结与解决方案

## 🐛 遇到的问题及解决方案

### 1. GitHub Pages 404 问题
**问题**：部署到 GitHub Pages 后访问子页面 404
**原因**：
- 路径配置错误，没有考虑 GitHub Pages 的子路径前缀
- 绝对路径 `/xxx` 在子路径下无法正确解析
**解决方案**：
- 所有链接使用相对路径 `./xxx` 或 `../xxx`
- Vite 配置 `base: './'` 使用相对路径构建
- 多层级页面使用正确的相对路径层级（比如 `../../apps/xxx`）

### 2. 上传功能 SHA 错误
**问题**：使用 `createOrUpdateFileContents` API 时报错 `"sha" wasn't supplied`
**原因**：
- 新增文件不需要提供 sha，但更新现有文件必须提供
- 批量新增文件使用底层 Git Tree API 更合适
**解决方案**：
- 新增文件直接使用 Tree API，不需要提供 sha
- 简化上传逻辑，只新增不修改现有文件，避免 sha 问题

### 3. 应用广场数据为空
**问题**：应用广场页面没有显示任何应用
**原因**：
- GitHub API 访问限制，匿名请求频率受限
- 动态加载逻辑复杂，容易出错
**解决方案**：
- 改为纯静态 HTML，直接把应用列表写在页面中
- 移除所有动态加载逻辑，确保 100% 显示内容
- 新增应用时手动添加 HTML 代码，简单可靠

### 4. 分类页面链接错误
**问题**：俄语专区、游戏专区内的链接跳转 404
**原因**：
- 相对路径层级错误，`../` 层级不够
- 二级目录下的页面需要使用 `../../` 跳转到根目录的应用
**解决方案**：
- 所有子分类页面的链接统一使用 `../../apps/xxx/index.html` 格式
- 统一路径规则，避免层级错误

### 5. 系统页面和用户应用混淆
**问题**：应用广场把系统分类页面（游戏专区、俄语专区）也当成应用展示
**原因**：
- 自动扫描没有过滤系统页面
**解决方案**：
- 纯静态列表手动维护，只添加真正的用户应用
- 分类页面不加入应用广场列表

### 6. GitHub API 权限问题
**问题**：后端 API 调用 GitHub 时权限不足
**原因**：
- GitHub Token 权限不够
- Token 没有 repo 权限
**解决方案**：
- 生成 Token 时勾选完整的 repo 权限
- 不要给 Token 多余的权限，确保安全

### 7. Vercel 部署 Node.js 版本问题
**问题**：Vercel 提示 Node.js 20 废弃
**解决方案**：
- 不需要指定 Node.js 版本，使用 Vercel 默认版本即可
- 移除 vercel.json 中的 runtime 配置

---

## ✅ 最佳实践总结
1. **优先使用静态方案**：能静态实现的功能不要用动态逻辑，稳定性更高
2. **统一路径规则**：所有链接使用相对路径，避免绝对路径
3. **简化逻辑**：复杂的动态功能如果不稳定，改为手动维护，可靠性更高
4. **最小权限原则**：GitHub Token 只给必要的权限
5. **分层设计**：系统分类页面和用户应用分开管理，避免混淆
